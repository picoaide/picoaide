# 聊天易用性改进报告与实施计划

## 一、用户旅程全景分析

### 1.1 完整用户旅程地图

```
首次进入               发送消息              等待响应             查看结果             后续操作
  │                     │                    │                   │                   │
  ▼                     ▼                    ▼                   ▼                   ▼
┌─────────┐   ┌─────────────────┐   ┌───────────────┐   ┌──────────────┐   ┌──────────────┐
│ 空状态   │   │ 输入文本 → 发送  │   │ 沙箱启动 5-15s│   │ 流式渲染     │   │ 继续对话     │
│ "开始第  │   │                 │   │ (无任何反馈)   │   │ Markdown     │   │ (无重试/编辑) │
│ 一次对话" │   │ 单行 input      │   │               │   │ 工具调用显示  │   │              │
└─────────┘   └─────────────────┘   │ 或直接出结果   │   │ 思考折叠显示  │   │ 或离开       │
                                    └───────────────┘   └──────────────┘   │ (回来加载    │
                                                                           │  历史)       │
                                                                           └──────────────┘
```

### 1.2 各阶段痛点明细

#### 阶段一：进入聊天

| # | 问题 | 严重度 | 现状 |
|---|------|--------|------|
| P1 | 空状态无引导 | 高 | 纯文本 "开始你的第一次对话吧！" |
| P2 | 加载中只有文字 | 中 | "加载中..." 文本，无骨架屏 |
| ~~P3~~ | ~~新建对话~~ | — | ~~不需要，单会话模式~~ |

#### 阶段二：发送消息

| # | 问题 | 严重度 | 现状 |
|---|------|--------|------|
| P1 | 单行 input，不能换行 | **极高** | `<input type="text">`，粘贴长文本只能横滚 |
| P2 | 无 Shift+Enter 换行 | 高 | 只有 Enter 发送 |
| P3 | 发送后无 loading 反馈 | 高 | 按钮切换但无"发送中"状态指示 |
| P4 | 不能粘贴图片/文件 | 中 | 纯文本输入 |

#### 阶段三：等待响应（最严重）

| # | 问题 | 严重度 | 现状 |
|---|------|--------|------|
| P1 | **沙箱启动期完全无反馈** | **极高** | 首次消息后 5-15 秒白屏，用户以为没反应 |
| P2 | 工具执行无进度指示 | 高 | 只显示一行文字，不知道工具在跑还是卡住 |
| P3 | 长时间无响应无法判断是否正常 | 中 | 没有 ETA、没有 spin 动画 |
| P4 | 失败时用户不知道发生了什么 | 高 | 错误直接嵌入聊天框，没有明确的错误提示 |

#### 阶段四：查看结果

| # | 问题 | 严重度 | 现状 |
|---|------|--------|------|
| P1 | **代码块无法复制** | **极高** | 用户手动选中 → 右键复制，技术用户每天用 |
| P2 | 消息不能复制 | 高 | 只能手动选中文本 |
| P3 | 滚动行为反直觉 | 高 | 新消息强行滚到底部，用户上翻看历史时被干扰 |
| P4 | 无消息时间戳 | 中 | 不知道消息是何时发的 |
| P5 | 连续消息不合并 | 低 | 每条消息独立气泡，视觉碎片化 |
| P6 | 无 "已停止" 提示外的结束状态 | 低 | 完成/错误/中断状态不明显 |
| P7 | 工具调用结果截断 60 字符，无法展开查看 | 低 | 完整结果对用户隐藏 |

#### 阶段五：后续操作

| # | 问题 | 严重度 | 现状 |
|---|------|--------|------|
| P1 | **失败消息无法重试** | **极高** | 错误后只能手动重新输入完整消息 |
| P2 | 消息无法编辑 | 高 | 打错字不能改，只能重发 |
| P3 | 无法重新生成 AI 回答 | 高 | 不满意只能重新问 |
| P4 | 离开后回来工具调用丢失 | 中 | 历史重放时工具状态行消失 |
| P5 | 无 "清空对话" 功能 | 中 | 只能刷新页面 |
| P6 | 无搜索历史消息 | 低 | 想看之前的内容只能手动滚动 |

#### 阶段六：AI 操作透明度

| # | 问题 | 严重度 | 现状 |
|---|------|--------|------|
| P1 | **子代理启动/运行完全不可见** | **极高** | LLM 调用了 subagent_spawn 后前端无任何反馈，用户以为没反应 |
| P2 | **LLM 重试时用户不知情** | 高 | LLM 调用失败自动重试 3-5 次，用户看到的是"卡死" |
| P3 | **上下文压缩时无提示** | 中 | 压缩过程同步阻塞 5-30 秒，界面完全冻结 |
| P4 | **停止按钮是否生效不确定** | 中 | cancelCtx 从 Background 派生，极端情况下停止信号不生效 |
| P5 | 子代理的思考/工具调用不与前端共享 | 高 | 子代理内部所有事件隔离在 goroutine 内，不经过父引擎的 cb 回调 |

## 二、本版本改进目标

### 2.1 范围

- **不切换模型** — 固定模型，服务器配置
- **不增加参数调节** — 无 Temperature/Top-P 等
- **不支持文件上传** — 纯文本聊天
- **不考虑暗色模式** — 保持当前风格
- **不考虑移动端适配** — 仅桌面端
- **不考虑多会话管理** — 单会话模式

### 2.2 核心原则

1. **无脑可用** — 用户输入文字 → 看到回复，没有学习成本
2. **容错** — 出错了能重试，不用重新打字
3. **可回溯** — 历史消息可查阅、可复制
4. **有反馈** — 每个操作都有明确的视觉反馈

### 3.7 AI 操作透明度

#### F19：子代理状态推送

当前子代理的整个生命周期对前端不可见。需要建立从子代理到前端的事件通道。

改进方案：
- 子代理启动 → 前端显示 "🔀 子代理「查资料」已启动，正在独立运行..."
- 子代理运行中 → 定期推送进度摘要（由子代理 engine 通过父 engine 的 cb 转发）
- 子代理完成 → 显示 "✅ 子代理「查资料」完成"

```
🔧 使用 subagent_spawn(name="查资料")
┌─ 🔀 子代理「查资料」正在运行 ──────┐
│  🤔 正在搜索信息...              │
│  🔧 调用 search_web              │
└──────────────────────────────────┘
✅ subagent_spawn → 子代理「查资料」已后台启动

... 稍后 ...

✅ 子代理「查资料」完成，共 2846 字
```

**后端变更**：
- `subagent.go` 子代理 engine.Process 的 callback 增加事件转发机制，将关键事件（text_delta 摘要、tool_call_start、finish、error）通过回调函数传回父引擎
- 父引擎 `engine.go` 将子代理事件包装为 `subagent_event` 类型发出
- 新增 SSE 事件类型 `subagent_event`

**前端变更**：`chat.js` 处理 `subagent_event` 类型，渲染子代理状态卡片

#### F20：LLM 重试提示

当前 LLM 调用失败时自动静默重试，用户看到的是"卡死了"。

改进：
- 重试时在前端显示 "⚠️ LLM 请求超时，正在重试 (2/5)..."
- 重试成功后恢复流式输出，用户感知不到中断

```
⚠️ LLM 请求超时，正在重试 (2/5)...  ───  几秒后消失
                                     ↘
                              正常流式输出继续
```

**后端变更**：`engine.go` LLM 重试循环中发出 `llm_retry` 事件
**前端变更**：`chat.js` 处理 `llm_retry` 事件，显示/隐藏重试提示

#### F21：压缩进度提示 + 解冻

当前上下文压缩同步阻塞，UI 完全冻结 5-30 秒。即使发送事件也没用——压缩是 CPU 绑定的同步代码，事件要等压缩跑完才被消费 goroutine 发送到浏览器。

改进：
- 压缩前后分别发出 `compressing` / `compressed` 事件
- 每次压缩循环迭代之间插入 `time.Sleep(10 * time.Millisecond)`，让 Go 调度器有机会切换到事件消费 goroutine
- 这样 `compressing` 事件能在压缩开始后立即送达浏览器

```
📦 正在压缩对话历史...
  → (每次压缩迭代间让出 10ms)
  → 浏览器收到事件，显示提示
  → 压缩完成后消失
```

**后端变更**：`engine.go` 压缩循环中每次压缩传递间插 `time.Sleep(10ms)` + 发事件
**前端变更**：`chat.js` 处理压缩事件，显示进度提示

#### F22：停止按钮可靠性

当前问题：`cancelCtx` 从 `context.Background()` 派生，取消信号通过 bridging goroutine 转发，存在竞态。

修复：
```go
// 改前
cancelCtx, cancelAll := context.WithCancel(context.Background())

// 改后
cancelCtx, cancelAll := context.WithCancel(ctx)
```

直接继承父 context，取消会自动传播，无需 bridging goroutine。

## 七、后端设计问题（影响用户体验）

### 7.1 分类说明

后端问题虽不直接可见，但最终都会让用户感知为"卡顿"、"没反应"、"丢了聊天记录"。

### 7.2 Critical 级别

| ID | 问题 | 文件 | 影响 |
|----|------|------|------|
| C1 | **旧 run 的 finish() 会删除新 run** | `chat_stream.go:121,157-165` | 连续发送两条消息 → 第二条的 run 被意外删除，`handleChatStop` 报"没有正在运行的对话" |
| C2 | **LoadArchive 死代码** | `session_io.go:82-97` | 历史 `archive.jsonl` 的降级分支永远不执行，升级后的用户历史为空 |
| ~~C3~~ | ~~客户端断开后沙箱继续跑~~ | — | **设计如此** — 用户关掉页面后沙箱应继续跑完，回来可看结果 |
| C4 | **沙箱启动无超时** | `chat_stream.go:161` | 启动阶段（overlay/网络/picoagent）卡住时用户永久等待。运行阶段由 AI 控制，不应设超时 |
| C5 | **记忆进化阻塞沙箱回收 30 秒** | `picoagent/main.go:332-363` | 对话结束后，下次发送要等 30 秒进化完成，用户以为卡死 |

### 7.3 Major 级别

| ID | 问题 | 文件 | 影响 |
|----|------|------|------|
| M1 | **用户锁并发释放** | `manager.go:139-312` | 两个沙箱可同时运行，资源冲突 |
| M2 | **事件写到错误会话目录** | `chat_stream.go:209-234` | 刷新后历史加载为空白 |
| M3 | **历史只读第一个会话** | `handlers.go:514` | 多个会话时只显示一个 |
| M4 | **IM 消息与 Web 消息竞态** | `integration.go:186-191` | IM 消息静默丢失 |
| M5 | **Engine cancelCtx 脱离父 context** | `engine.go:295-303` | 停止按钮有时不生效 |
| M6 | **压缩阻塞整个 Agent 循环 5-30s** | `engine.go:329-363` | 上下文压缩是 CPU 绑定同步代码，即使发事件也要等压缩完才送达浏览器，用户看到 UI 完全冻结 |
| M7 | **事件持久化竞态** | `chat_stream.go:97-109` | 异常退出时丢失事件 |

### 7.5 后端修复优先级（与前端配合）

#### 第一优先：P0 前端改动的前置依赖

| 后端问题 | 阻塞的前端功能 | 说明 |
|---------|---------------|------|
| C4 无启动超时 | F6 启动进度反馈 | 启动阶段设超时（30s），运行阶段不设限 |
| N2 无启动阶段事件 | F6 启动进度反馈 | 后端不发出阶段事件，前端无法显示进度 |
| C1 旧 run 删除新 run | F14 失败重试 | 重试需要可靠的 run 生命周期 |

#### 第二优先：直接影响体验

| 后端问题 | 影响 | 建议修复方案 |
|---------|------|-------------|
| M5 + N10 cancelCtx 脱离父 context | 停止按钮有时不生效 | 用 `context.WithCancel(ctx)` 替换，5 行 |
| C5 记忆进化阻塞 30s | 连续对话卡死 | 进化改为异步，不阻塞沙箱回收 |
| M6 压缩阻塞 UI | UI 冻结 5-30s | 压缩循环间插 `time.Sleep(10ms)` 让事件送达 + 发 progress 事件（F21） |
| N9 子代理事件隔离 | 前端看不到子代理运行（F19） | 子代理 callback 增加事件转发到父引擎 cb |

#### 第三优先：数据完整性

| 后端问题 | 影响 | 建议修复方案 |
|---------|------|-------------|
| C2 LoadArchive 死代码 | 历史丢失 | 修复 if 判断顺序 |
| M2 事件写到错误目录 | 历史空白 | 用会话 key 定位目录 |
| M3 只读第一个会话 | 多会话不可见 | 遍历所有会话目录 |
| M1 用户锁并发释放 | 双沙箱 | 用 atomic.Bool 保护 released |

### 7.6 建议本版本修复范围

**必须修（影响前端改动）：**
- C4 + N2 → F6 启动进度反馈的前置条件
- C1 → 重试功能的基础

**建议修（成本低、收益大）：**
